# Домашнее задание: JWT + Spring Security

## Тема

Практика по аутентификации и авторизации в Spring Boot приложении с использованием **JWT (JSON Web Tokens)** и **Spring Security**.

## Проект: «Подземелье драконов» (Dragon's Lair API)

Вам предстоит реализовать REST API для онлайн-игры в жанре текстового RPG, где игроки исследуют подземелье, сражаются с монстрами и собирают сокровища. Часть API доступна всем (например, просмотр бестиария и таблицы лидеров), а часть — только зарегистрированным игрокам, мастерам подземелий и администраторам.

> Если тема показалась скучной — можно заменить на свою, **но согласовать с преподавателем**. Главное, чтобы сохранились требования к функционалу (см. ниже). Примеры альтернатив: API для аукциона лотов, сервис бронирования квестов, мини-соцсеть для мемов, бэкенд для покерного клуба и т.п. Тудулисты и онлайн-библиотеки — **запрещены**.

## Требования к функционалу

### 1. Регистрация и аутентификация

- `POST /api/auth/register` — регистрация нового игрока (логин, пароль, email).
- `POST /api/auth/login` — вход, возвращает **access token (JWT)** и **refresh token**.
- `POST /api/auth/refresh` — обновление access-токена по refresh-токену.
- Пароли должны храниться в БД в **хешированном** виде (`BCryptPasswordEncoder`).
- JWT должен содержать `username`, `roles`, `exp`, `iat`.

### 2. Роли пользователей

Минимум **3 роли**:

| Роль          | Описание |
|---------------|----------|
| `ROLE_PLAYER` | Обычный игрок. Может сражаться, собирать лут, смотреть свой профиль. |
| `ROLE_MASTER` | Мастер подземелья. Может создавать новых монстров и подземелья. |
| `ROLE_ADMIN`  | Администратор. Может банить игроков, удалять любые сущности, выдавать роли. |

Управление ролями должно быть реализовано через `GrantedAuthority` и проверяться через `@PreAuthorize` или конфигурацию `SecurityFilterChain`.

### 3. Маршруты

#### Публичные (без авторизации)

- `GET /api/bestiary` — список всех монстров.
- `GET /api/bestiary/{id}` — карточка монстра.
- `GET /api/leaderboard` — топ-10 игроков по опыту.
- `POST /api/auth/**` — регистрация / вход / refresh.

#### Защищённые (только авторизованные)

- `GET /api/me` — профиль текущего игрока.
- `POST /api/dungeons/{id}/enter` — войти в подземелье.
- `POST /api/dungeons/{id}/attack` — атаковать монстра.
- `GET /api/inventory` — инвентарь игрока.

#### Только для `ROLE_MASTER` и выше

- `POST /api/bestiary` — создать монстра.
- `POST /api/dungeons` — создать подземелье.
- `PUT /api/bestiary/{id}` — отредактировать монстра.

#### Только для `ROLE_ADMIN`

- `DELETE /api/players/{id}` — забанить (удалить) игрока.
- `POST /api/players/{id}/roles` — выдать/забрать роль.
- `DELETE /api/bestiary/{id}` — удалить монстра.

### 4. Безопасность

- Все защищённые маршруты должны проверять валидность JWT через **кастомный фильтр** (`OncePerRequestFilter`).
- При невалидном/просроченном токене — статус `401 Unauthorized`.
- При недостаточных правах — статус `403 Forbidden`.
- CSRF можно отключить (REST API stateless).
- Сессии — `SessionCreationPolicy.STATELESS`.
- Глобальный `@ControllerAdvice` для обработки ошибок безопасности.

### 5. Доменная модель (минимум)

- `User` — id, username, password, email, roles, experience.
- `Role` — id, name.
- `Monster` — id, name, hp, attack, level, lootTable.
- `Dungeon` — id, name, difficulty, monsters.
- `Item` — id, name, rarity, ownerId.
